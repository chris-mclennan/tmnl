## Two tracks

tmnl has two modes. **Shell mode** is your daily driver — a fast GPU terminal hosting your `$SHELL`. **Native mode** is the differentiator — apps render directly into tmnl as structured cells, skipping ANSI entirely.

## Why "native mode" matters

If you've ever tried to render a real UI inside a terminal — buttons, knobs, fine-grained graphics — you've fought ANSI. tmnl's native mode is the escape: an app opens a Unix socket, sends `Frame { cells: [Cell { fg, bg, char, ... }] }` directly, and tmnl draws it on the GPU. No vt100 parser between you and the pixels.

[`mnml`](https://mnml.sh) (a terminal IDE) and [`mixr`](https://mixr.sh) (a terminal DJ app) both ship a `--blit <socket>` mode that connects them as native tmnl tabs. Inside tmnl, mixr's crossfader and knobs render as actual graphics, not unicode block art. mnml gets native-tab integration with workspace switching.

You don't have to use any of this. tmnl works fine as a regular terminal emulator hosting your shell. The cell-streaming layer just makes it the host for a class of apps that couldn't exist before.

## The family

tmnl is one of three small Rust tools by Chris McLennan:

- **tmnl** — this terminal
- **[mnml](https://mnml.sh)** — a NvChad-style terminal IDE (vim or standard editing, LSP, git, fuzzy everything, baked-in HTTP client)
- **[mixr](https://mixr.sh)** — a terminal DJ app (Beatport streaming, Claude DJ, MIDI controllers)

mnml and mixr both run inside any terminal. Inside tmnl specifically, they get native-tab integration and the cell-streaming aesthetic.
